### Visual Walkthrough 

When we look closely at each parking bay, the asphalt will have a small circular cutout, inside which a sealed grey enclosure with the PCB and a battery is kept. It is flush with the road so snowplows and tyyres never touch it. 

**When the Slot is Empty**

The board is inactive; STM32WLE5 is in Stop Mode 2 with CPU clock frozen and radio off. Even BMM350 is in in low-power state. The board only draws about 2 microamps which will allow the battery to last longer. The only two things that are active are the 32.768 kHz crystal and BMM350's interrupt logic waiting for magnetic disturbance.

**When a Car Stops**

The Earth's magnetic field, which was passing uniformly through the asphalt, will now bend and concentrate through the ferrous mass. The vector, perhaps, shifts by 10-50 µT.
The BMM350, still in its low-power state, notices the shift. Its internal comparator crosses the programmed threshold. It pulls its INT pin low — a digital shout to the sleeping STM32.

For this, interrupts (the peripheral sends a hardware signal when it needs attention, allowing the CPU to perform other tasks or stay in sleep-mode until then) are used instead of polling (method where the CPU continuously checks the status of a peripheral or input in a loop).

**Wakeup Sequence**

The falling edge on PB7 ripples through the STM32's pin synchroniser and the clock tree restarts. With it, CPU begins executing from the interrupt vector table. First, the firmware enables the I2C peripheral clock. Then it wakes the BMM350 with a command. It waits a few milliseconds for the 1.8 V rail to stabilise after the sudden load of the CPU waking up. This is beacuse the capacitors need time to charge. If the firmware rushes the I2C transaction while the supply is still drooping, the BMM350 might return garbage data. A few milliseconds of patience prevents a false transmission that would waste battery and airtime.

**I2C Conversation**

Two wires come alive: SCL (PB6) and SDA (PB8). The STM32 sends a START condition — SDA drops while SCL is high, the universal "attention please" signal. It addresses the BMM350. The sensor acknowledges by pulling SDA low for one clock tick.
The STM32 requests the data registers. The BMM350 replies with 6 bytes: two for X, two for Y, two for Z. Each byte is clocked out on SDA, sampled by the STM32 on the rising edge of SCL. The STM32 acknowledges every byte except the last one, then sends STOP.
The bus goes quiet. The pull-up resistors (R4, R5) gently hold both lines high again, ready for the next conversation.

Also, each axis is 16 bits. The firmware needs the full 3D vector to compute magnitude, which is orientation-independent. A single-axis reading might miss a car aligned unfavourably with the sensor's sensitive direction. This is why we have 6 bytes.

**Decision Making**

The firmware converts the raw bytes to signed integers and computes:

ΔB=∣B<sub>measured</sub>∣ − ∣B<sub>baseline</sub>∣ 

The baseline was learned during installation, when the bay was known empty. The delta is large — well above the hysteresis threshold. The state machine flips from VACANT to OCCUPIED. Bit 0 of the payload becomes 1.

**Building the Packet**

The firmware enters the LoRaWAN stack. It checks its non-volatile memory: does it have valid session keys from a previous OTAA join? Yes. It constructs the MAC frame:

- **MHDR**: Uplink, confirmed (this state change matters since if this packet is lost, the dashboard still shows green (vacant) but a car is actually sitting there)
- **DevAddr**: the short address derived from the Device EUI
- **FCtrl**: ADR enabled, ACK requested
- **FCnt**: 1427 (increments with every transmission, prevents replay attacks)
- **FPort**: 1 (application data)
- **FRMPayload**: 1 byte. Bit 0 = 1 (occupied)
  
The payload is encrypted with AppSKey. The MIC is computed with NwkSKey and appended.

Because the air is public. Anyone with a cheap USB radio dongle can listen to 865 MHz. Encryption ensures that even if they capture the packet, they cannot tell whether the particular bay is occupied or vacant. It also prevents an attacker from forging a "vacant" packet to steal a reserved slot.

**Radio**

The STM32's internal LoRa modem wakes. It configures:
- Frequency: 865.2 MHz (India ISM band)
- Spreading Factor: SF10 (conservative; ADR may lower it later)
- Bandwidth: 125 kHz
- TX Power: 14 dBm

The internal DC-DC converter (using inductor L2) powers the power amplifier efficiently. The firmware drives PA11 to select RF1 — the chip antenna path. The modem modulates the chirp: a frequency sweep that carries the encrypted packet.
The signal leaves the chip, passes through the matching network (L3, C15, C18, L4, L5...), through the RF switch, and radiates from the ceramic antenna. The chirp lasts perhaps 200–400 milliseconds. Then the PA shuts down. The SMPS shuts down. Every microsecond of radio-on time is a tax on battery life.

**Gateway**

Half a kilometre away, on a terminal roof, a LoRaWAN gateway listens. Its multi-channel receiver is scanning 865–867 MHz continuously. It catches the chirp, demodulates it, wraps the raw bytes in a UDP/IP packet, and forwards it over Ethernet to the ChirpStack server.

**Server**

ChirpStack receives the packet. It checks the MIC using NwkSKey: valid. It looks up the DevAddr: registered to "T3-Bay-0xx". It decrypts the payload using AppSKey. It deduplicates — if two gateways heard the same chirp, only one copy is kept.
The JavaScript codec runs: `bytes[0] & 0x01` extracts the occupancy bit. The JSON is published to MQTT:
```json
{
  "deviceName": "T3-Bay-0xx",
  "occupied": 1,
  "time": "2026-06-18T12:30:00Z"
}
```

**Dashboard Turns Red**

The parking dashboard subscribes to the MQTT topic. It sees occupied: 1 for Bay 4. The tile flips from green to red. A timestamp updates. Perhaps a small battery icon shows 82%.
A facility manager glances at the screen. the particular bay is taken. No action needed. The system worked.

**ACK Returns**

Back underground, the STM32 opened a receive window 1 second after transmission. The gateway's ACK arrives — a tiny downlink packet confirming receipt. The firmware marks the frame as acknowledged.
If the ACK had been lost, the firmware would increment a retry counter and attempt retransmission on the next event. But this time, all is well.

**Return to Sleep Sequence**

The firmware reverses the wake-up sequence:
- Put BMM350 back into low-power mode via I2C command.
- Disable the I2C peripheral clock.
- Clear the PB7 interrupt flag and re-arm it for the next magnetic event.
- Set the RTC alarm for the next heartbeat — perhaps 6 hours from now.
- Enter Stop Mode 2. The CPU freezes. The radio dies. Only the RTC and the interrupt pin remain alive.
