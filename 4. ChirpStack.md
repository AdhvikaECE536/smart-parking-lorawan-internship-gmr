ChirpStack is open-source software. GMR hosts it on a private server or cloud VM. It is the central post office of the LoRaWAN system.

### Authentication and Integrity Check

ChirpStack receives the raw packet bytes. Before it even looks at the payload, it performs two checks:
1. Device Lookup: The packet header contains a DevAddr (device address). ChirpStack checks its database: is this DevAddr registered under the "Smart-Parking-T3" application? If not, the packet is dropped silently.
2. MIC Verification: Using the NwkSKey stored for this device, ChirpStack recomputes the Message Integrity Code and compares it to the one in the packet. If they do not match, the packet is discarded (tampered or corrupted).

---

### Deduplication

Airport parking lots may have multiple gateways. If three gateways here the same chirp from the same Bay, they forward each chirp copy to ChirpStack.

ChirpStack sees three identical packets. It keeps one and discards the duplicates. It may record metadata (which gateways heard it, signal strength from each) for coverage analysis, but the application only receives one event.

---

### Decryption
The packet passes the MIC check. ChirpStack now uses the AppSKey to decrypt the FRMPayload.
Out comes a small byte array — often just 1 or 2 bytes:
- Byte 0: Bit 0 = 1 (occupied) or 0 (vacant)
- Byte 1 (optional): Battery level or temperature


It it decrypted here and not in the gateway since it might be owned by a third-party network operator. End-to-end encryption ensures only the application owner sees the parking status.

### Payload Decoding (The Codec)

Raw bytes are meaningless without context. ChirpStack runs a small **JavaScript decoder** (configured per device profile) to translate the byte array into human-readable JSON.

**Example conceptual decoder:**
```javascript
function Decode(fPort, bytes) {
  var occupied = bytes[0] & 0x01;      // Extract bit 0
  var battery = bytes[0] >> 1;         // Maybe bits 1-7 encode battery
  return {
    occupied: occupied,                // 1 or 0
    battery_percent: battery * 2        // Example scaling
  };
}
```

 A codec lets you adapt without changing the server source code. It is a plugin system for payload formats.

After decoding, ChirpStack produces a JSON object:
```json
{
  "deviceName": "T3-Bay-047",
  "devEUI": "a8...b1",
  "occupied": 1,
  "battery_percent": 82,
  "rxInfo": [
    { "gatewayID": "t3-rooftop-01", "rssi": -87, "snr": 8.5 }
  ],
  "fCnt": 1427,
  "time": "2026-06-18T12:30:00Z"
}
```

### Pushing to Dashboard

ChirpStack does not draw parking maps. It pushes the JSON to an external system via an **integration**.

##### MQTT Integration (Most Common)
ChirpStack publishes to a topic like:
```
application/123/device/a8...b1/event/up
```

The dashboard backend (Node-RED, Python Flask, or a custom web app) subscribes to this topic. The moment the JSON arrives, the backend updates its database and refreshes the UI.

**Why MQTT?**  
Because it is designed for real-time event streams. The alternative — polling a REST API every second — wastes bandwidth and adds latency. With MQTT, the dashboard updates the instant the car parks.

##### HTTP Integration (Alternative)
ChirpStack POSTs the JSON to a webhook URL. This is simpler if the dashboard backend already exposes a REST receiver, but it is less efficient for high-frequency events.

### Dashboard

The dashboard receives the JSON. It looks up the device EUI in its own database:
- `a8...b1` → Bay 47, Terminal 3, Zone B

It updates the tile:
- `occupied: 1` → **Red icon** (or a filled car symbol)
- `occupied: 0` → **Green icon** (or an empty slot symbol)
- `battery_percent: 82` → Small battery icon (green/yellow/red depending on threshold)


We use only one bit for occupancy because the dashboard already knows this is Bay 47. The payload only answers the yes/no question. This minimalism is what makes the system scalable to thousands of bays without flooding the network.

## Alerts & Monitoring

ChirpStack or the downstream dashboard is typically configured with alerts:

| Condition | Alert | Why it matters |
|-----------|-------|----------------|
| No heartbeat > 12 hours | "Bay 47 sensor may be dead" | A dead sensor looks like a permanently vacant slot. Revenue loss. |
| Battery byte < 20% | "Replace battery in Bay 112" | Prevents surprise failures. Maintenance can schedule a visit. |
| Gateway offline > 10 min | "T3-Rooftop-01 down" | Affects hundreds of bays. Urgent. |
| FCnt anomaly | "Possible replay attack" | Security flag. Investigate immediately. |

## Provisioning a New Sensor (Engineer Workflow)

1. **Install sensor** in Bay 47. Power on.
2. **Add device** in ChirpStack: enter Device EUI, App EUI, App Key.
3. **Join:** Sensor sends OTAA Join Request. ChirpStack accepts. Session keys are generated.
4. **First Uplink:** Sensor sends heartbeat. Appears in Live Frames.
5. **Verify decoder:** Engineer checks that `occupied: 0` appears when bay is empty.
6. **Test:** Park a car. Watch Live Frames for `occupied: 1` (confirmed). Check ACK arrives.
7. **Dashboard link:** MQTT pushes data. Bay 47 tile turns red.
8. **Production:** Device is marked active. Monitoring alerts are enabled.
